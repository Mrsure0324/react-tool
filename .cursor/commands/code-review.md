## 审查维度

逐个维度检查，每个维度给出 ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修复：

### 1. 正确性（最重要）
- 逻辑是否正确
- 边界情况是否处理（空值、空数组、极端输入）
- 异步操作是否安全（竞态、取消、错误处理）
- 数据类型是否安全

### 2. 性能
- 不必要的重渲染（缺少 memo / useCallback / useMemo）
- 大列表是否使用虚拟化
- 是否有内存泄漏（未清理的 listener / timer / subscription）
- 网络请求是否有防抖/节流

### 3. 可维护性
- 命名是否自解释
- 函数/组件是否单一职责
- 嵌套是否过深（> 3 层要提前返回或拆分）
- 是否有魔法数字或硬编码字符串

### 4. 规范一致性
- 是否符合项目现有的代码风格（参考 code-style 规则）
- 样式是否使用 LESS 变量，不硬编码颜色值
- 类型定义是否完整（无 any）
- 导入是否使用 `@/` 路径别名

### 5. 安全性
- XSS / 注入风险
- 敏感数据是否暴露（token、密钥）
- 用户输入是否校验
